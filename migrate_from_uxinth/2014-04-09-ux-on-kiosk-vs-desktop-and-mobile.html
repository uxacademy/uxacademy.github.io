จากบทที่แล้วเราเน้นกันเรื่องการมอง ความคาดหวังของผู้ใช้ เป็นหลักแทนที่จะดูแค่ลักษณะของอุปกรณ์ โดยยกตัวอย่างพฤติกรรมและความคาดหวังของผู้ใช้บน Mobile และ Tablet ว่าแตกต่างกัน ทีนี้พอมามอง KIOSK จะต่างออกไปอีกมากเลยครับ

KIOSK ส่วนมากจะตั้งอยู่ในที่สาธารณะ มีหน้าที่ให้ข้อมูลหรือให้ผู้ใช้ทำธุรกรรมต่างๆ ตั้งแต่ซื้ออาหาร จ่ายเงินจนถึงแลกบัตรส่วนลด หน้าที่ของ KIOSK แต่ละตัวจะต่างกันไป แต่มักจะมีสิ่งที่เหมือนกันคือเครื่อง KIOSK ไม่ต้องการให้ผู้ใช้อยู่นานๆ เพราะจะมีคนต่อคิวใช้งานอยู่ แม้แต่ตัวผู้ใช้เองก็ไม่อยากอยู่นานเมื่อทำงานเสร็จก็ต้องการไปแล้วล่ะครับ

ด้วยเงื่อนไขนี้ทำให้ผู้ใช้มีความคาดหวังต่อโปรแกรมบนเครื่องต่างจากโปรแกรมบน Desktop และ Mobile ครับ ผมลองแจงรายละเอียดของเงื่อนไขออกมาเป็นข้อๆ ได้ดังนี้ครับ

  1. เราเจอผู้ใช้ครั้งเดียว ไม่ต้องหวังให้ผู้ใช้เรียนรู้ ลองนึกถึงความคาดหวังของผู้ใช้ตอนเดินไปที่เคาร์เตอร์ธนาคาร หรือเคาร์เตอร์จ่ายเงินบัตรเครดิต ผู้ใช้ไม่ได้คาดหวังว่าจะต้องเรียนรู้อะไร สิ่งที่ทำคือตอบคำถามพนักงาน พอเดือนต่อมาก็ลืมไปหมดแล้วว่าพนักงานเคยถามอะไรบ้าง

    ดังนั้นโปรแกรมบน KIOSK จึงไม่ควรให้คนเรียนรู้อะไรเลย ถ้าตัดเรื่อง Tab เรื่องรูป icon เรื่องการวาง control ต่างๆ ออกไปจะดีมากๆ เพราะสิ่งเหล่านี้สร้างภาระให้ผู้ใช้ต้องเรียนรู้ทั้งสิ้น (อย่างน้อยก็ควรตัดออกจาก flow หลักของโปรแกรม) สิ่งที่ต้องรำลึกไว้เสมอคือผู้ใช้ไม่สนใจจะจำ หรือทำความเข้าใจกับลำดับการทำงานของโปรแกรมของเราเลย ทำงานจบก็จบกัน ไว้ครั้งหน้าค่อยเริ่มใหม่ ต่างจากโปรแกรมบน Desktop และ Mobile ที่ผู้ใช้มีเวลาทำความเข้าใจกับระบบ เพื่อให้ครั้งต่อไปเค้าทำงานได้เร็วขึ้น

    ใครที่ออกแบบโปรแกรมบน Desktop และ Mobile มานานๆ ต้องระวังให้ดีครับ สามารถพลาดได้ง่ายๆ

  2. ถามทีละอย่าง ทีนี้ลองจิตนาการว่าพนักงานที่เคาร์เตอร์ยื่นกระดาษมาให้เรากรอก แล้วบอกว่าเราที่เดียว 5 ช่องว่าต้องกรอกอะไรบ้าง สิ่งที่มักจะตามมาทันทีในหัวผู้ใช้คือ จำไม่ได้ ตามมาด้วย อยากให้บอกทีละช่อง แม้ว่าจริงๆ แล้วแค่อ่านบนกระดาษก็พอจะรู้แล้วว่าต้องกรอกอะไรบ้าง แต่ก็ไม่อยากอ่านอยู่ดี

    โปรแกรมของเราก็เหมือนกันครับ ถ้าในหนึ่งหน้ามีเรื่องต้องทำมากกว่าหนึ่งอย่าง ผู้ใช้จะเริ่มรู้สึกว่ายากทันที ต่างจากการกรอกฟอร์มที่บ้าน หรือการใช้งานโปรแกรม Desktop หรือ Mobile ที่ผู้ใช้มีความสะบายใจที่จะศึกษา

    การเอาหลายอย่างมาไว้ในหน้าเดียวกันไม่ใช่ภาระ แต่เป็นการลดงาน ไม่ต้องกดไปหลายหน้า

  3. อย่าให้เดา วินาทีที่มีคนมาต่อข้างหลัง หรือเวลาที่อยู่ในที่สาธารณะ สิ่งที่ผู้ใช้ไม่อยากทำมากที่สุดคือ ทำผิด ดังนั้นอะไรก็ตามที่ช่วยลดความผิดพลาดได้ ให้ทำไว้ก่อนเช่น อย่าใส่แต่ icon ให้ผู้ใช้เดา ต้องใส่ label ไว้เสมอ เป็นต้น

    ใครที่เดาได้หรือดู icon แล้วรู้ก็สามารถกดได้ทันที ส่วนคนที่ไม่แน่ใจเค้าจะอ่าน label และคนที่ไม่แน่ใจมากๆ ถ้าเรามีคำอธิบายให้เค้าสั้นๆ ด้านข้างจะดีมากๆ

  4. วาง UI สำหรับคนที่เรื่องมากไม่เท่ากัน ข้อ 4 จะต่อจากข้อ 3 คือมีข้อมูลแต่ก็อย่าให้ทุกอย่างเด่นเท่ากันหมด เพราะจะทำให้โปรแกรมของเราดูรกและผู้ใช้ก็ไม่รู้ว่าควรดูอะไรก่อน

    • ผู้ใช้ที่ไม่คิดมาก พอเห็น icon เดาได้ก็จะกดเลยดังนั้นควรให้ icon เด่นสุด
    • ผู้ใช้ที่คิดมากนิดหน่อย จะอยากอ่าน label ก่อนแล้วค่อยกด ตรงนี้มีงานวิจัยชัดเจนกว่าปุ่มที่มี label มีคนกดมากกว่า (อย่าพึ่งเชื่อนะครับ ผมยังหา paper ไม่เจอว่าเคยอ่านมาจากไหน)
    • ผู้ใช้ที่คิดมาก นอกจากอ่าน label แล้วอาจจะยังไม่พอใจ เค้าจะอ่านคำอธิบายของปุ่มเพิ่มเติมว่ากดแล้วมันจะทำอะไร ส่วนมากแค่บอกเพิ่มเติมอีกนิดหน่อยก็พอใจแล้ว
    • ผู้ใช้ที่วิตกกังวล โดยเฉพาะโปรแกรมด้านการเงิน จะอยากอ่านคำอธิบายที่ชัดเจนเพิ่มเติมก็ควรมีให้เค้าครับ แต่ส่วนนี้อาจจะเป็นตัวหนังสือจางๆ เลยก็ได้ เพราะคนกลุ่มนี้มีพลังความกังวลมากพอที่จะพยายามอ่านครับ
  5. ผู้ใช้รู้ว่าต้องการทำอะไรอย่างชัดเจนก่อนจะเดินไปถึงเครื่อง ต่างจากโปรแกรมบน Desktop และ Mobile ที่ผู้ใช้ไม่จำเป็นต้องรู้ว่าตัวเองต้องการอะไร แค่โปรแกรมดูน่าสนใจ ดูเหมือนว่าจะแก้ปัญหาให้เราบางอย่างก็ลอง Download มาเล่นดู เพราะต้นทุนในการทดลองต่ำ ต่างจาก KIOSK ที่ผู้ใช้ต้องเดินทางไปที่ตู้

    ดังนั้นผู้ใช้จึงคาดหวังให้มันทำงานตามที่ต้องการอย่างมากซึ่งแทบเป็นไปไม่ได้ที่จะออกแบบ UI ให้ตรงกับที่คาดหวัง ดังนั้นเราจึงทำตรงกันข้ามโดยใช้เทคนิคของพนักงานเคาร์เตอร์ครับ ถามให้ผู้ใช้ตอบที่ละคำถาม แล้วงานของผู้ใช้จะเสร็จก่อนที่เค้าจะรู้ตัวซะอีก

  6. อย่าเชื่อสิ่งที่ผู้ใช้คิด ให้ดูว่าเค้าทำอะไร ข้อสุดท้ายเป็นเรื่องของการทำ Usability Test ครับ ในการใช้งาน KIOSK เรื่องนี้จะสำคัญเป็นพิเศษ เพราะปกติแล้วการตัดสินใจของมนุษย์เราระหว่างมีเวลาคิดกับไม่มีเวลาคิด ผลลัพธิ์จะต่างกันมาก โดยเฉพาะ KIOSK ที่มีเวลาคิดน้อยเป็นพิเศษ การทดสอบยิ่งต้องดูว่าผู้ใช้ทำอะไร อย่าพึ่งรีบเชื่อคำแนะนำของผู้ใช้หรือถ้าเราถามว่ามันควรเป็นอย่างไร ก็อย่าพึ่งรีบเชื่อคำตอบของผู้ทดสอบครับ

ทั้ง 6 ข้อผมสรุปออกมาโดยอาศัยประสบการณ์หลักจากการทำ User Research ในกระบวนการออกแบบโปรแกรมบนเครื่อง TrueMoney KIOSK ซึ่งสุดโต่งไปทางด้านความเร็วในการจบรายการ ดังนั้นอาจจะต่างไปจาก KIOSK อื่นๆ ก็ขอให้ลองประยุกต์กันดูนะครับ
