ต่อจากบทความแรกที่เค้าทดลองปรับ UI ของ Gmail เวอร์ชั่น Mobile Web ซึ่งโจทย์ค่อนข้างท้าทาย เพราะเค้าต้องการสร้างความเป็นตัวของตัวเอง และให้มีความเรียบง่ายไปด้วย ซึ่งหลายข้อจะไปซ้ำกับ Apple Mail ทำให้ต้องหลบกันหลายท่า มาตอนที่สองจะมีอีกสองข้อที่น่าสนใจครับ

ข้อที่ 2 คือ การสื่อสารกับผู้ใช้ต้องให้เหมือนๆ กันหมด

ตัวอย่างที่ยกมาคือลูกศรที่เป็นคำสั่งเพิ่มเติม ในตัวอย่างด้านบน (Message screen) เมื่อกดลูกศรแล้วชุดคำสั่งจะยาวออกมา แล้วลูกศรจะเปลี่ยนจากชี้ลงเป็นชี้ขึ้นแทน แต่ในการเลือก account ลูกศรกลับไม่เปลี่ยนไปเมื่อผู้ใช้กดไปแล้ว

อันนี้แก้ไม่ยากครับ ทีมงานต้องตกลงกันว่าจะให้หัวลูกศรชี้กลับขึ้นไปด้านบนหรือเปล่า แล้วก็ทำให้เหมือนกันหมดเท่านั้นเอง

ข้อที่ 3 ทำให้เหมือนกับโปรแกรมอื่นๆ

หลังจากที่ผู้ใช้เริ่มใช้มือถือ เค้าจะได้เรียนรู้พฤติกรรมต่างๆ ของโปรแกรมที่มากับเครื่อง เช่น Apple's Mail หรือ Safari Browser เป็นต้น ผู้ใช้ก็จะคาดหวังว่าโปรแกรมอื่นๆ จะมีพฤติกรรมไปในแนวทางเดียวกัน โดยเฉพาะกับความสามารถที่ช่วยให้ผู้ใช้สบายขึ้น ผู้ใช้จะคาดหวังสิ่งเหล่านี้โดยอัตโนมัติ

Gmail จะไม่เหมือนโปรแกรมอื่นๆ ในสองจุด อันดับแรกคือ ส่วนของ popover หรือหน้าต่างที่เปิดขึ้นมาเพื่อให้ผู้ใช้เลือก เช่นตอนดูคำสั่งเพิ่ม หรือตอนที่เลือก account ผู้ใช้จะคาดหวังว่าเมื่อกดนอกหน้าต่างที่ popover ขึ้นมาแล้ว หน้าต่างนั้นจะหายไป แต่ไม่เป็นแบบนั้นบน gmail ผู้ใช้จะต้องกดปุ่มเดิมซ้ำเท่านั้นจึงจะปิดไปได้

อันดับที่สองคือการ zoom เข้าและออก สำหรับ Apple's Mail เมื่อ double tab บนเนื้อหาที่สนใจ มันจะ zoom ออกมาเต็มหน้าจอตามรูป  แต่บน Gmail มันจะไม่สนใจ double tab ของเราเลย

จริงๆ เป็นเรื่องเล็กๆ น้อยๆ นะครับ แต่มันก็กระทบกับภาพใหญ่ของ user experience ได้เลย ดังนั้นในเรื่องความสม่ำเสมอถ้าเราต้องการจะทำให้มันเปลี่ยนไป จะต้องทดสอบกับผู้้ใช้ให้ดี ถ้าผลตอบรับออกมาว่าแบบของเรานั้นดีกว่าก็ค่อยทดลองนำมาใช้ครับ

สุดท้ายจริงๆ แล้วมีเหตุผลมากมายที่ทำให้โปรแกรม Gmail ออกแบบมาเป็นอย่างที่เราเห็นกัน ทั้งเหตุผลที่เหมาะสมและเหตุผลแปลกๆ แต่อย่างไรก็ตามผู้ใช้ก็ไม่ได้คิดเหมือนเราทุกคน ดังนั้นเราจึงควรมุ่งประเด็นไปที่การทำให้โปรแกรมของเราใช้งานได้ แก้ปัญหาของผู้ใช้ได้ มากกว่าการทำให้มันดูสวยงาม ไม่ใช่ว่าความสวยงามไม่จำเป็นนะครับแค่มันเป็นรองแค่นั้นเอง ^_^
