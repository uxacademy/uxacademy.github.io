ตอนที่หนึ่งเราพูดถึงปัญหาเว็บไซต์ที่ยาวมากๆ จนผู้ใช้ไม่สามารถจำโครงสร้างได้ทั้งหมดเพราะปัญหา short-term memory ส่งผลต่อการเข้ามาครั้งที่สองหรือการย้อนกลับไปอ่านส่วนที่สนใจ แต่ก็ใช่ว่าจะไม่มีใครเหมาะกับเว็บยาวๆ เลย มีเว็บและโปรแกรมหลายแบบที่เหมาะกับโครงสร้างยาวๆ ครับ

ตอนที่สองเราพูดถึงทางแก้เมื่อเว็บของเรายาวมากๆ เราจะย่อมันลงได้อย่างไร โดยที่ผู้ใช้ และผู้บริหารจะยังคงชอบมันอยู่

ตอนที่สามเราจะพูดถึงกรณีที่เหมาะกับการให้ข้อมูลจำนวนมากตั้งแต่หน้าแรก เพราะเว็บยาวๆก็มีตลาดของมันเอง

กรณีแรกที่เหมาะสำหรับเว็บยาวๆ คือจะต้องเป็นเว็บที่ผู้ใช้ไม่ต้องการย้อนกลับมาอ่านอีกครั้ง ส่วนมากจะเป็นเว็บข่าวที่มีข่าวใหม่มาทุกวัน ผู้ใข้อ่านหนึ่งรอบแล้วจบไม่ต้องย้อนกลับมาอ่านอีก เพราะข่าวทั้งหมดได้เปลี่ยนไปแล้วการอ่านรอบสองจึงก็ไม่ต่างจากอ่านรอบแรก

 
ตัวอย่างเว็บ Thairat และ Facebook

เจ้าของเว็บต้องการอำนวยความสะดวกให้ผู้ใช้สามารถอ่านจากบนลงล่างได้หมดไม่จำเป็นต้องรู้ภาพรวมของเว็บไซต์ กรณีของเว็บแบบนี้เราจะไม่พยายามเปิดหน้าใหม่ ถ้าจะดูรายละเอียดอาจจะเปิดหน้าใหม่ หรือทำเป็น popup บนเว็บเดิม และที่สำคัญคือเนื้อหาควรเปลี่ยนทุกวันเพราะผู้ใช้จะต้องย้อนมาอ่านใหม่หมดทุกครั้ง

ลองคิดดูว่าถ้าเว็บแบบนี้เข้ามาครั้งที่สองแล้วไม่ต่างจากเดิม การเข้ามาครั้งที่สามผู้ใช้จะไม่คาดหวังว่าจะมีจุดต่าง

กรณีที่สองคือโปรแกรมที่โครงสร้างและลำดับความสำคัญของเนื้อหาเปลี่ยนบ่อย เช่น เว็บข่าว หรือเว็บกิจกรรมของบริษัท หรือเว็บ portal เพราะเว็บลักษณะนี้มักจะมีข่าวสำคัญ หรือการจัดกิจกรรมที่ไม่สามารถจัดลำดับความสำคัญได้ชัดเจน จนต้องปรับขนาดของเนื้อหาอยู่ตลอด

flipboard ปรับโครงสร้างตามเนื้อหา

flipboard เป็นตัวอย่างของ App ที่เปิดยาวไปเรื่อยๆ คล้ายกับที่เราเห็นจากนิตยสาร หรือหนังสือพิมพ์จริงๆ แต่มักไม่เห็นแบบนี้กับบนเว็บไซต์เพราะการปรับตำแหน่งเนื้อหาเป็นงานที่แพงมากสำหรับเว็บไซต์ (แต่เว็บข่าวใหญ่ๆ บางทีก็คุ้มที่จะเปลี่ยน หรือเตรียมให้พร้อมสำหรับการเปลี่ยน) หรือถ้าเป็นโปรแกรมก็ต้องออกแบบมาให้ปรับเปลี่ยนโครงสร่างได้ง่าย หรืออัตโนมัติไปเลย

กรณีที่สามคือเว็บที่ออกแบบมาให้อ่านครั้งเดียว เช่นเว็บไซต์ประชาสัมพันธ์กิจกรรม เว็บไซต์แนะนำบริษัท หรือเว็บที่นำเสนอแนวคิดบางอย่างที่เมื่อผู้ใช้เข้าใจแล้วก็ไม่ต้องกลับมาอ่านอีก

ตัวอย่างหน้าเว็บของ Springfield MO และ Free Range Designs

เมื่อก่อนเว็บพวกนี้มักสร้างด้วยโปรแกรม flash เพื่อให้ดูหวือหวาดึงดูความสนใจเพิ่มโอกาสรับรู้ข้อมูลมากขึ้น แต่เดี๋ยวนี้ก็เริ่มมีพวกเว็บ HTML5 ที่ออกแบบมายาวๆ ดูแล้วเพลินดีแต่ให้ดูหลายรอบคงไม่ไหว

ตอนนี้นอกจากงานออกแบบทั้งสามแนวนี้แล้ว ผมยังนึกไม่ออกว่ามีเว็บหรือโปรแกรมไหนที่ควรให้ข้อมูลผู้ใช้มากๆ ตั้งแต่หน้าแรก เพราะโปรแกรมหรือเว็บส่วนมากต้องให้ผู้ใช้สามารถกลับมาใช้ซ้ำได้บ่อยๆ

แม้แต่เว็บข่าวที่มีข่าวสารที่ออกทุกวัน เดี๋ยวนี้ยังพยายามออกแบบโครงสร้างให้เข้าใจได้ง่าย พยายามรวบกล่องข้อมูลหลายกล่องให้รวมอยู่ในกล่องเดียว แล้วใช้ tag, category หรือสี เพื่อช่วยในการแยกประเภทแทนที่จะแยกกล่องออกไป

เว็บ blognone.com

จากภาพตัวอย่าง จะเห็นว่าเว็บนี้ออกแบบมาแบบที่ให้ข้อมูลผู้ใช้มากๆ ตั้งแต่หน้าแรก แต่ก็สามารถทำให้ผู้ใช้เข้าใจโครงสร้างได้อย่างรวดเร็ว โดยใช้เทคนิคการรวบเนื้อหาเป็นกล่องยาวๆ กล่องเดียว แล้วใช้สีและ icon เพื่อแยกประเภท การออกแบบลักษณะนี้ผู้ใช้จะสามารถจดจำโครงสร้างได้อย่างรวดเร็วครับ

เรื่องความจำระยะสั้นนอกจากจะใช้กับการออกแบบ UI แล้ว ถ้าเราตีรวม programmer ที่เข้ามาอ่าน code ของเราเป็นหนึ่งใน user เราก็จะไม่เขียน code ยาวๆ ที่รวมอยู่ใน method เดียวเพราะคนที่เข้ามาอ่าน code ก็จะจำได้ไม่หมด หรือกรณีของ graphic designer ถ้าเราคิดว่าคนที่เข้ามาออกแบบต่อจากเราเป็น user หรือแม้แต่ตัวเราเองที่ต้องย้อนกลับมาแก้งานออกแบบของตัวเองเป็น user เราก็จะแยก layer ให้เป็นกลุ่มๆ ชัดเจน ไม่รวมมาอยู่ใน group เดียว

นั่นก็เพราะเราคำนึงถึง user และเราจึงต้องศึกษาข้อจำกัดต่างๆ เอาไว้ และความจำก็เป็นหนึ่งในปัจจัยสำคัญครับ
